# 3D Gaussian Splatting 技术分享

---

# 第一部分：3DGS 技术解析

## 一、3DGS 是什么

3D Gaussian Splatting（3DGS）是 2023 年 SIGGRAPH 发表的一项 3D 场景表示与渲染技术。它将三维场景表示为**数百万个带有颜色和透明度的高斯椭球**，通过将这些椭球从 3D 投影到 2D 屏幕来完成渲染。

与传统方案的核心区别在于：**只需要拍照片，就能重建出可实时渲染的 3D 场景，无需美术手工建模**。

---

## 二、核心原理

### 2.1 高斯球（Gaussian）

每个高斯球包含以下属性：

| 属性 | 说明 |
|------|------|
| 位置（xyz） | 高斯球在 3D 空间中的坐标 |
| 旋转（四元数） | 控制椭球的朝向 |
| 缩放（3 轴标准差） | 控制椭球在每个轴的大小 |
| 颜色（球谐函数系数） | 实现视角相关的颜色变化 |
| 透明度（opacity） | 控制高斯球的可见程度 |

> 可以类比为"一个会随视角变色的彩色雾气泡泡"。

### 2.2 Splatting（投影）

将每个 3D 高斯椭球投影到屏幕上，得到一个 2D 高斯斑（Splat）。核心过程是将 3D 协方差矩阵投影到 2D 视平面，最终每个 Splat 就是一个**带有颜色和透明度渐变的椭圆**。

### 2.3 Alpha Blending（混合叠加）

多个高斯球在同一像素上叠加时，必须**从近到远排序**后按 alpha-compositing 公式叠加。这个**深度排序**是整个渲染实现中最大的性能瓶颈。

## 三、与传统 3D 渲染方式的对比

### 3.1 三种方案横向对比

| 维度 | Mesh（传统网格） | NeRF（神经辐射场） | 3DGS（高斯泼溅） |
|------|------------------|-------------------|------------------|
| 诞生时间 | 计算机图形学早期 | 2020 年 | 2023 年 |
| 场景表示 | 三角面片 + 纹理贴图 | 隐式神经网络 | 高斯椭球点云 |
| 建模方式 | 美术手工建模 | 照片重建 | 照片重建 |
| 渲染速度 | 实时 | 慢（逐像素查询 MLP） | 实时（快 NeRF 10-100 倍） |
| 浏览器运行 | 支持（WebGL/Three.js） | 基本不可行 | **可以实时运行** |
| 视觉质量 | 依赖模型精度和贴图 | 高（照片级真实感） | 高（照片级真实感） |

### 3.2 优势

- **制作成本低**：拍照即可重建，不需要专业 3D 建模师
- **渲染速度快**：相比 NeRF 快 10-100 倍，可在浏览器实时运行
- **视觉质量高**：照片级真实感，能表达毛发、烟雾等复杂质感
- **工程友好**：数据结构简单（点云 + 属性），易于压缩和传输
- **端上可用**：WebGL/WebGPU 均可支持，移动端也能运行

### 3.3 劣势

- **编辑困难**：高斯球没有拓扑结构，很难做任意形变和部件替换
- **动态场景支持弱**：人物运动、流水、烟雾等动态内容需要扩展方案（4DGS），文件体积极大
- **大场景工程复杂**：城市级别需要分块加载 + LOD，工程复杂度高
- **文件体积大**：原始 `.ply` 文件动辄 500MB-2GB，需要额外压缩
- **反光/透明材质不佳**：纯反光和高透明材质的重建效果有限
- **移动端性能受限**：高斯球数量需控制在 30 万以下才能保证 30fps

## 四、浏览器渲染实现要点

### 4.1 文件格式

- **`.ply`**：原始格式，每个高斯球约 59 个 float 字段，体积大
- **`.splat`**：压缩格式，每个高斯球 32 字节
- **`.ksplat`**：进一步压缩，每个高斯球约 19 字节（约 12:1 压缩比）

```js
// 读取 .splat 文件，每个高斯球 32 字节
const f32 = new Float32Array(buffer, offset, 3);   // xyz position
const u8  = new Uint8Array(buffer, offset + 24, 8); // scale(3) + rot(4) + opacity
```

### 4.2 深度排序优化

每帧相机移动后，所有高斯球需要重新按深度排序（几十万到几百万个）。关键方案：

- **Web Worker**：排序放到 Worker 线程，避免阻塞主线程
- **基数排序（Radix Sort）**：替代 JS 内置 `Array.sort`，速度快 10 倍以上
- **SharedArrayBuffer**：零拷贝传递排序结果

```js
// Worker 内部：基数排序 + 每帧通过 SharedArrayBuffer 传递结果
const sortedIndices = radixSort(depths);
self.postMessage({ sortedIndices }, [sortedIndices.buffer]);
```

### 4.3 WebGL 渲染

用四边形（quad）作为 instance，每帧更新排序后的 draw order：

```glsl
// fragment shader 核心
vec2 d = vPos;
float alpha = exp(-0.5 * dot(d, d)) * vOpacity;
if (alpha < 1.0/255.0) discard;
gl_FragColor = vec4(vColor * alpha, alpha);
```

### 4.4 WebGPU 未来方向

WebGPU 可以在 GPU 上直接做排序（Compute Shader），省去 CPU 和 GPU 之间的数据回传，Chrome 113+ 已正式支持。

### 4.5 性能优化一览

| 优化手段 | 效果 |
|----------|------|
| Worker + Radix Sort | 排序耗时从 ~80ms 降到 ~8ms |
| SharedArrayBuffer 零拷贝 | 省去 Uint32Array 的 structuredClone 开销 |
| 视锥剔除（Frustum Culling） | 减少 20-40% 的排序量 |
| 降低高斯球数量（剪枝） | 数量砍半，帧率接近翻倍 |
| 使用 `.ksplat` 格式 | 加载速度提升 5-10 倍 |
| WebGPU Compute Shader 排序 | GPU 排序比 CPU 快 3-5 倍 |

## 五、前端可用工具库

| 库 | 特点 | 适用场景 |
|----|------|----------|
| `@mkkellogg/gaussian-splats-3d` | 基于 Three.js，API 友好，GitHub 最活跃 | 已有 Three.js 项目集成 |
| `gsplat.js`（PlayCanvas 出品） | 更轻量，WebGPU 优先 | 性能敏感场景 |
| Luma AI WebGL Component | `<luma-ai-viewer>` Web Component，零代码 | 快速原型验证 |

快速上手示例（20 行代码即可跑起来）：

```js
import { Viewer } from '@mkkellogg/gaussian-splats-3d';
const viewer = new Viewer({
  cameraUp: [0, -1, 0],
  initialCameraPos: [0, 1, 3]
});
viewer.addSplatScene('./scene.ksplat').then(() => viewer.start());
```

## 六、值得关注的技术进展

- **Compact 3DGS**：向量量化压缩，进一步缩小文件体积
- **Gaussian Opacity Fields**：支持从高斯球中提取 Mesh
- **4DGS / Dynamic 3DGS**：支持动态场景
- **Gaussian Splatting + NeRF 混合方案**：结合两者优势

# 第二部分：3DGS 在电商中的落地方案

## 一、电商应用场景

### 1.1 3D 商品展示

**核心价值**：替代传统 3D 建模流程，拍照即可生成商品 3D 模型，大幅降低成本。

**适合品类**：
- 鞋靴、箱包：硬质材质，重建效果好
- 家具、家居饰品：静态物体，尺寸适中
- 珠宝配饰：高光细节丰富，视角变化的真实感是卖点
- 3C 数码：外观精致，消费者注重细节

**不太适合的品类**：
- 服装（布料形变、穿着效果无法很好呈现）
- 食品生鲜（保质期内外观变化大）

### 1.2 场景化展示

- **家具 + 室内场景**：消费者可在还原的真实房间中浏览家具摆放效果
- **线下门店数字化**：将线下旗舰店/体验店用 3DGS 重建，线上虚拟逛店

### 1.3 商品详情页增强

在传统图片/视频基础上，增加 3D 交互查看入口，用户可自由旋转、缩放商品，提升转化率。

---

## 二、完整落地流程

### 2.1 采集阶段

```
手机/相机绕物体拍 100-200 张照片
     ↓
关键要求：
  - 照片重叠率 60% 以上
  - 光照均匀，避免强阴影
  - 避免纯反光/纯白/纯透明材质
  - 覆盖所有角度（包括顶部和底部）
```

**大规模采集方案**：可搭建简易摄影棚 + 转台，标准化拍摄流程，降低人工成本。

### 2.2 重建阶段

```
照片 → COLMAP（SfM 稀疏重建） → 官方 3DGS 训练 → 原始 .ply 文件
```

| 方案 | 要求 | 耗时 | 成本 |
|------|------|------|------|
| 本地训练（官方仓库） | NVIDIA GPU | 小场景 30 分钟，大场景 2-4 小时 | 硬件成本 |
| 云端服务（Polycam / Luma AI） | 上传照片即可 | 10-30 分钟 | 按次/按月付费 |
| 自建训练平台 | GPU 集群 | 可并行批量处理 | 基础设施投入 |

### 2.3 后处理阶段

**清理**：使用 `SuperSplat` 等工具可视化清理飘散噪点高斯球

**压缩**：
- `.ply` → `.ksplat`：约 12:1 压缩比
- 典型商品模型压缩后约 5-20MB

**质量检查**：人工抽检重建质量，确保无明显伪影

### 2.4 前端集成

```
┌──────────────┐    ┌───────────────┐    ┌──────────────┐
│  CDN / OSS   │ →  │  前端 Viewer   │ →  │  商品详情页   │
│ 存储 .ksplat │    │ Three.js 渲染  │    │  嵌入 3D 组件 │
└──────────────┘    └───────────────┘    └──────────────┘
```

**关键技术决策**：

| 决策点 | 推荐方案 |
|--------|----------|
| 渲染库 | `@mkkellogg/gaussian-splats-3d`（Three.js 生态） |
| 文件存储 | CDN + OSS，就近分发 |
| 加载策略 | 渐进式加载（先加载低精度版本，后台加载完整版） |
| 移动端适配 | 高斯球数量控制在 30 万以下，降采样渲染 |
| 降级方案 | 不支持 WebGL 2.0 的设备回退到图片/视频 |

### 2.5 流式加载策略

```
用户打开页面
     ↓
加载低精度版本（稀疏高斯球，~1MB）→ 快速呈现大致效果
     ↓
后台继续加载完整版（~10MB）→ 平滑过渡到高质量渲染
```

类似渐进式 JPEG 的体验，用户无需等待完整加载即可交互。

## 三、电商场景下的性能优化策略

| 场景 | 策略 |
|------|------|
| 首屏加载 | 首帧用预渲染图片占位，3D 加载完成后切换 |
| 低端设备 | 动态降低高斯球数量，降低渲染分辨率 |
| 流量控制 | 用户主动点击"查看 3D"才加载，避免自动加载浪费带宽 |
| 内存管理 | 离开视口时释放 GPU 资源，切换商品时复用 Viewer 实例 |
| 移动端 | iOS Safari 暂不支持 WebGPU，需 WebGL 回退方案 |

## 四、落地挑战与应对

| 挑战 | 应对方案 |
|------|----------|
| 大规模商品采集成本 | 标准化摄影棚 + 转台，批量自动化训练 |
| 文件体积影响加载速度 | `.ksplat` 压缩 + CDN 加速 + 渐进式加载 |
| 移动端性能不足 | 分级渲染 + 低端设备降级为图片/视频 |
| 部分品类效果差 | 明确适用品类白名单，不强推不适合的场景 |
| 动态展示需求 | 短期用多角度静态 3DGS，长期关注 4DGS 进展 |
| ROI 衡量 | A/B 测试对比 3D 展示 vs 传统图片的转化率差异 |

## 五、分阶段落地建议

### Phase 1：技术验证（1-2 周）

- 选择 1-2 个典型商品，完成拍照 → 重建 → 前端展示的全流程
- 验证压缩比、加载速度、渲染帧率等关键指标
- 确定渲染库选型

### Phase 2：小规模试点（2-4 周）

- 选择一个品类（如家具），完成 20-50 个商品的 3DGS 制作
- 在商品详情页上线 3D 查看功能
- 收集用户行为数据（停留时长、交互率、转化率）

### Phase 3：规模化生产（1-3 个月）

- 搭建标准化采集和训练流水线
- 建设内部 3D 资产管理平台
- 扩展到更多品类

### Phase 4：体验升级（持续迭代）

- AR 试穿/试摆（结合 WebXR）
- 多商品场景组合展示
- 关注 4DGS 等新技术的落地可能性

## 六、总结

3DGS 为电商 3D 化提供了一条**低成本、高质量、可实时渲染**的技术路径。相比传统 3D 建模，它将商品 3D 展示的门槛从"需要专业建模团队"降低到了"拍照即可"。

核心建议：

1. **从适合的品类切入**：优先选择硬质、静态、消费者关注外观细节的商品
2. **渐进式上线**：先用低精度版本保证体验，再用高精度版本提升质量
3. **做好降级方案**：确保不支持 WebGL 的设备仍有良好体验
4. **用数据说话**：通过 A/B 测试验证 3D 展示对转化率的实际影响
